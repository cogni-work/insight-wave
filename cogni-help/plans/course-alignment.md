# Service Plan: Align cogni-help Courses with docs/workflows

**Status**: Proposal
**Branch**: `claude/cogni-help-course-alignment-0TknQ`
**Owner**: cogni-help maintainers
**Plugin**: cogni-help (currently v0.0.5, Incubating)

## 1. Problem

cogni-help is the meta-layer that teaches users to operate the insight-wave
ecosystem. It ships **two parallel surfaces** that should reinforce each other but
have drifted apart:

- **`teach` skill** — 12 interactive courses under `skills/teach/references/courses/`
- **`workflow` skill** — 6 cross-plugin pipeline templates under `skills/workflow/references/workflows/`

Meanwhile, the canonical user-facing tutorials live in **`docs/workflows/`** at
the workspace root (7 documents, generated by `cogni-docs`). These three sets do
not agree on names, scope, or pipeline shape — so a learner who completes a
course rarely lands on the matching workflow guide, and a user who reads
`docs/workflows/` cannot find the corresponding course.

### 2.2 Concrete drift findings

1. **Name divergence on the same content** — `consulting-engagement` vs
   `new-engagement`; `research-to-report` vs `research-to-slides`;
   `trends-to-solutions` vs `trend-to-marketing`. Only `portfolio-to-pitch`
   shares a name, and even there the pipelines disagree (`docs/` describes a
   single-plugin path through `portfolio-communicate`; the cogni-help template
   describes a four-plugin chain via narrative/sales/visual).
2. **Three docs workflows have no plugin equivalent** —
   `install-to-infographic`, `portfolio-to-website`, `content-pipeline`. A user
   typing `/workflow` cannot discover them.
3. **Two plugin templates have no docs equivalent** — `docs-pipeline`,
   `full-onboarding`. They are correct internal playbooks but invisible to
   anyone reading `docs/workflows/`.
4. **Plugin coverage gaps in courses** — `cogni-website` and `cogni-wiki` are
   not taught in any course, even though both feature in canonical workflows
   (`portfolio-to-website`, `research-to-report`). `cogni-help`'s own `guide`,
   `troubleshoot`, `cheatsheet`, `cogni-issues`, and `course-deck` skills are
   referenced in Course 1 only at the meta level.
5. **Course → workflow handoff is missing** — Of the 12 courses, only
   Course 12 ends with a `/workflow <name>` pointer. The other 11 end with a
   plugin-guide link but never invite the learner into the pipeline that turns
   their newly-learned plugin into a deliverable.
6. **Pipeline shape in `cogni-help/skills/workflow/research-to-slides.md` is
   stale** — it omits `cogni-claims` (verification) and `cogni-copywriting`
   (polish), both of which are mandatory steps in the `docs/` canonical
   `research-to-report` workflow. Following the cogni-help template produces a
   weaker deliverable than following the docs.
7. **README / CLAUDE.md mention "5 cross-plugin workflow playbooks"** — the
   directory actually contains 6, and `README.md` lists 6 (drift inside
   cogni-help's own metadata).

## 3. Goal

A learner who completes a cogni-help course knows exactly which canonical
`docs/workflows/` tutorial turns that plugin into a deliverable, and the
in-plugin `workflow` template names + pipelines agree with the docs. The
12-course curriculum covers every plugin that appears in a canonical workflow.

## 4. Proposed Changes

### 4.1 Rename + re-shape workflow templates to match `docs/workflows/`

| Action | Current | Target |
|---|---|---|
| Rename | `workflow/references/workflows/new-engagement.md` | `consulting-engagement.md` |
| Rename + re-shape | `research-to-slides.md` | `research-to-report.md` (insert wiki → research → **claims** → **copywriting** → visual) |
| Rename + re-shape | `trend-to-marketing.md` | `trends-to-solutions.md` (replace `marketing` step with `visual` story-to-slides / enrich-report) |
| Re-shape | `portfolio-to-pitch.md` | Match the `portfolio-communicate use_case=pitch` end-to-end flow from docs |
| Add | — | `install-to-infographic.md` (mirror docs canonical) |
| Add | — | `portfolio-to-website.md` (mirror docs canonical) |
| Add | — | `content-pipeline.md` (mirror docs canonical, including narrative bridge for long-form) |
| Keep | `docs-pipeline.md` | Keep as cogni-docs internal playbook; add note "internal — no docs/ equivalent" |
| Keep | `full-onboarding.md` | Keep as meta-onboarding (cogni-help's own scope) |

End state: 9 workflow templates in cogni-help — 7 mirroring `docs/workflows/`,
plus `docs-pipeline` and `full-onboarding` as internal.

### 4.2 Course → workflow handoff (template change in every course)

Add a uniform "Run this next" block to each course's **Course Completion**
section, sitting between **Reference documentation** and **Next recommended
course**:

```markdown
**Run this workflow next**:
- `/workflow <name>` — in-session playbook
- `docs/workflows/<name>.md` — full tutorial with diagrams and prerequisites
```

### 4.3 Close plugin-coverage gaps in the curriculum

Two minimally-invasive options. Recommend **Option B** as the smaller change.

**Option A — Add two courses (13–14)**:
- Course 13: `cogni-wiki` — knowledge base management for research compounding
- Course 14: `cogni-website` — portfolio-driven static site generation

**Option B — Extend existing courses (recommended)**:
- Course 7 (Visual Deliverables) gains a final module: **"From slides to a
  multi-page site (cogni-website handoff)"** — covers theme inheritance and
  `portfolio-to-website` entry points.
- Course 8 (Research Reports) gains a Module 0: **"Wiki-backed research
  (cogni-wiki primer)"** — explains how `cogni-wiki` stores knowledge that
  feeds `cogni-research`. Aligns Course 8 with the canonical
  `research-to-report` pipeline that begins at the wiki.

Option B keeps the "12 courses, ~9 hours" promise in the README intact while
restoring full plugin coverage.

### 4.4 Fix internal metadata drift

- `cogni-help/CLAUDE.md` line 20 — change "5 cross-plugin workflow playbooks"
  to match the actual count after 4.1 (9).
- `cogni-help/CLAUDE.md` and `README.md` — regenerate the **Workflow Templates**
  table to mirror the renamed/added templates.
- `cogni-help/skills/workflow/SKILL.md` — update any inline workflow list to
  match the new naming.
- Bump `cogni-help/.claude-plugin/plugin.json` from `0.0.5` → `0.0.6` (patch
  bump per CLAUDE.md's "after any change to skills, agents, or structure"
  rule). Mirror the bump in `.claude-plugin/marketplace.json`.

### 4.5 Add a maintainer guard

Add `cogni-help/scripts/check-workflow-alignment.sh`:

```
For each file in docs/workflows/*.md:
  assert a same-named file exists in cogni-help/skills/workflow/references/workflows/
  except: install-to-infographic, portfolio-to-website, content-pipeline if absent → warn
For each course in cogni-help/skills/teach/references/courses/*.md:
  assert "Run this workflow next" block is present and references an existing template
```

Output JSON in the standard `{"success": bool, "data": {...}, "error": "..."}`
shape. This is the cogni-help-side analogue of `cogni-docs/doc-audit` for the
courses ↔ workflows seam.

## 5. Out of Scope

- Rewriting any `docs/workflows/*.md` content. The canonical docs are the
  reference; cogni-help adapts to them.
- Adding new courses beyond what 4.3 Option B requires.
- Touching cogni-docs generators. If course-aware sections are needed in
  `docs/`, file a separate `cogni-docs` issue.
- Translating courses or workflows. Multilingual support stays a downstream
  concern.

## 7. Acceptance Criteria

- `ls cogni-help/skills/workflow/references/workflows/` produces the 7 names
  from `ls docs/workflows/` plus `docs-pipeline.md` and `full-onboarding.md`.
- Every workflow that exists in both places agrees on pipeline shape (same
  plugins, same order). Spot-check by diffing the pipeline line at the top of
  each pair.
- `grep -L "Run this workflow next"
  cogni-help/skills/teach/references/courses/*.md` returns no files.
- `cogni-help/scripts/check-workflow-alignment.sh` exits 0.
- The README and CLAUDE.md workflow tables match the directory listing.
- `cogni-portfolio`, `cogni-website`, `cogni-wiki`, `cogni-claims`, and
  `cogni-copywriting` each appear in at least one course AND at least one
  canonical workflow that the curriculum links to.

## 8. Risks & Open Questions

- **Pipeline-shape changes in `research-to-report` / `trends-to-solutions`
  templates may surprise users mid-session.** Mitigation: keep the renamed
  files as thin redirects to the new ones for one release if needed.
- **`portfolio-to-pitch` shape disagreement is genuine, not just naming.** The
  docs version uses the new `portfolio-communicate` end-to-end skill; the
  cogni-help template predates it. Decision needed: adopt the docs shape
  (recommended, since docs are canonical) or fork into
  `portfolio-to-pitch-classic` for the older 4-plugin chain.
- **Course extensions in 4.3 Option B push Course 7 and Course 8 past
  ~45 minutes.** Acceptable if the new module is marked optional; otherwise
  shift to Option A.
- **No service-plan tooling found in the repo.** `/service-plan` was referenced
  in the task description but no skill or command by that name exists locally.
  This document is the manual analogue; if a `service-plan` skill is
  introduced later, regenerate this plan through it for consistency.
