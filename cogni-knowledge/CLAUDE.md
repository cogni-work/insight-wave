# cogni-knowledge — developer guide

Wiki-first research orchestrator. Binds a `cogni-wiki` knowledge base to N research projects so the work compounds across runs instead of dying in chat history. The v0.1.0 inverted-pipeline path (Phases 1–7, plus the optional fail-soft **Phase 4.5** distillation interphase added at v0.1.13/#336) forks dedicated agents under `agents/` and runs zero-network claim verification, with no cogni-research dispatch in the runtime path — it is the only live path. The legacy v0.0.x chain (`knowledge-research` / `knowledge-report`) was archived under `_archive/` at M11 (v0.0.27). `agents/` and `scripts/` are populated accordingly.

## Architecture

```
.cogni-knowledge/binding.json   ← the only new state cogni-knowledge owns
       │
       ▼
knowledge-* skills (orchestrators)
       │
       ▼
cogni-wiki / cogni-research (delegate targets)
```

A "knowledge base" is one directory that contains both:

- `.cogni-wiki/config.json` — the wiki manifest (owned by `cogni-wiki`)
- `.cogni-knowledge/binding.json` — the binding manifest (owned by this plugin)

They live as siblings. The wiki is the substrate; the binding records which research projects have contributed.

## Wiki output layout

The inverted pipeline deposits its knowledge into `wiki/` as a **curated, progressively-disclosed** tree (`schema_version` 0.0.8) — a single curated front door over per-type machine-owned sub-indexes, not a flat dump:

```
wiki/
├── index.md            ← curated portal front door: per-theme map linking the
│                          sub-indexes below, with an intro that points at the
│                          overview narrative.
├── overview.md         ← machine-owned narrative home (MACHINE-OWNED:OVERVIEW-
│                          NARRATIVE), maintained in place by knowledge-finalize.
├── concepts/index.md   ← per-type sub-index (rendered by concepts_index.py)
├── sources/index.md    ┐
├── questions/index.md  │
├── syntheses/index.md  │ per-type machine-owned sub-indexes
├── entities/index.md   │
├── summaries/index.md  │
├── learnings/index.md  ┘
└── meta/               ← visible control files: log.md, context_brief.md,
                           open_questions.md
```

`schema_version` 0.0.8 is **additive and read-forward** — it declares the curated layout on top of the existing per-type-directory contract, exactly as the 0.0.6 (`sources/`) and 0.0.7 (`questions/`) bumps did. **0.0.5 stays the hard-fail boundary** (pre-migration wikis abort). This is the wiki `schema_version`, distinct from the cogni-knowledge plugin version (`plugin.json`).

The end-state this contract points at folds the overview narrative *into* the `index.md` intro, but that fold requires redirecting `knowledge-finalize`'s `overview_update.py` write target from `wiki/overview.md` to `wiki/index.md` — a separate follow-up child of the curated-layout epic. Until it lands, **`wiki/overview.md` stays the machine-owned narrative home** and `index.md` is a portal that links to it (this is what `knowledge-setup` seeds for a new wiki); `index.md` becomes the true narrative front door when the redirect child ships.

Two contract notes the rest of the layout work depends on:

- **Per-type `wiki/<type>/index.md` is a machine-owned sub-index, not a page** — generated, never authored, so it is **exempt from the `entries_count`, `orphan_page`, and `reverse_link_missing` checks**, parallel to the existing `is_audit_slug` (`lint-*` / `health-*`) exemption. The lint/health enforcement of this exemption lands in a follow-up child.
- **`wiki/meta/` is the declared target** for the visible control files (`log.md`, `context_brief.md`, `open_questions.md`). The actual path centralization — with a legacy fallback — is a separate follow-up; until it lands, the legacy flat `wiki/context_brief.md` / `wiki/open_questions.md` paths remain valid. Layout *declaration* (this contract) and layout *seeding* / *path move* are deliberately separate children of the curated-layout epic.

## Data model — `binding.json`

```json
{
  "knowledge_slug": "<kebab-case>",
  "knowledge_title": "<human readable>",
  "wiki_path": "<absolute path to wiki root>",
  "research_projects": [
    {
      "slug": "<research-project-slug>",
      "deposited_at": "<YYYY-MM-DD>",
      "report_path": "<absolute path to output/report.md>",
      "report_source": "web | local | wiki | hybrid",
      "project_path": "<absolute path to project root>"
    }
  ],
  "refresh_candidates": [
    {
      "synthesis_slug": "<wiki/syntheses/<slug> a newer source may have outdated>",
      "synthesis_title": "<title — the refresh topic surfaced to the user>",
      "triggered_by_source": ["<slug of each newly-ingested source that flagged it>"],
      "detected_at": "<YYYY-MM-DD>",
      "via_pages": ["<overlapping cited slug(s) — source/concept/entity>"],
      "status": "open"
    }
  ],
  "topic_lineage": {
    "covered_themes": [
      {
        "theme_key": "<normalized token-set key>",
        "question_slug": "<wiki/questions/<slug> this theme persists as>",
        "labels": ["<theme_label as phrased per run>"],
        "first_seen": "<YYYY-MM-DD>",
        "last_seen": "<YYYY-MM-DD>"
      }
    ],
    "open_themes": []
  },
  "curator_defaults": {
    "max_candidates_per_sq": 12,
    "score_threshold": 0.5,
    "fetch_cache_max_age_days": 30
  },
  "research_defaults": {
    "market": "dach",
    "output_language": "en",
    "prose_density": "standard",
    "tone": "objective",
    "citation_format": "ieee",
    "target_words": 4000
  },
  "charter": {
    "domain": "<one sentence: what this knowledge base is about>",
    "audience": "<primary reader of syntheses from this base>",
    "scope": "<geography / segment / horizon — in/out boundaries, one line>",
    "framed_at": "<YYYY-MM-DD, or '' when the base was never steered>"
  },
  "created": "<YYYY-MM-DD>",
  "schema_version": "0.1.5"
}
```

`topic_lineage.covered_themes[]` (entry shape defined at schema 0.1.3, #409) couples question-node accumulation across runs: each entry records one recurring research theme as `{theme_key, question_slug, labels[], first_seen, last_seen}`, where `theme_key` is `_knowledge_lib.theme_norm_key(theme_label)` (an order- and stopword-independent token-set built on `tokenize` — deliberately NOT `norm_key`, which would drop the `GENERIC_DENYLIST` tokens that distinguish themes like "AI Act" vs "AI System"). `knowledge-ingest` Step 4.5 reads it (via `question-store.py emit --binding`) so a variant `theme_label` routes to its existing `question_slug` node, and writes it (via `knowledge-binding.py upsert-themes`, the **single writer** of this block). Pre-0.1.3 bindings carry `covered_themes: []` (the `cmd_init` default), so every consumer reading `.get("topic_lineage", {}).get("covered_themes", [])` falls straight through. `open_themes` stays a plain string list; it is the **seed-theme backlog** the charter interview seeds at `knowledge-setup` (schema 0.1.4, see below) and the candidate menu for the first research question — distinct from `covered_themes[]` (the closure side written by `upsert-themes`). It is **write-once-at-init**: the stored list is never pruned, so it is the durable record of what the base set out to research. The open→covered **display lifecycle (#450, v0.1.64)** is computed at *read time* by `knowledge-binding.py themes` — it hides a seed whose `theme_norm_key` matches a researched `covered_themes[].theme_key` so the `knowledge-resume` / `knowledge-dashboard` backlog tracks reality instead of drifting stale. The pruning is **display-only**; `open_themes[]` itself is never mutated (approach (a) of #450 — non-destructive, the issue's stated preference over mutating the backlog at finalize).

`charter` (schema 0.1.4) records the base-level steering captured at `knowledge-setup` Step 2.5 — `{domain, audience, scope, framed_at}` (what the base is about, who reads its syntheses, its in/out boundaries). It is the coarse counterpart to `knowledge-plan`'s per-research-question topic-framing: one base accumulates many topics over time, so the base needs a charter while each run keeps the finer per-question framing (`references/charter-framing.md` is the two-level model + base-scoped question bank). The charter feeds forward as **inherited grounding** — `knowledge-plan` Step 0.4 reads `charter.{domain,audience,scope}` and anchors the per-question framing to it. All four fields default to `""` so a flag-only / non-interactive / `--no-charter` `init` still writes a complete block; `framed_at` is stamped only when any of domain/audience/scope is non-empty (so a default-skeleton charter is distinguishable from a real one). The charter is **domain/audience/scope/themes only** — the writer-quality knobs stay per-run on `knowledge-plan`. Consumers MUST read it with `.get("charter", {}).get(key, "")` so a pre-0.1.4 binding falls straight through; `knowledge-resume` / `knowledge-dashboard` surface it read-only.

`refresh_candidates` (schema 0.1.5) is the **evidence-aware** half of the refresh signal — the complement to the time-based staleness `lint_wiki.py` flags. When `knowledge-ingest-source` deposits a new source related to a synthesis's cited evidence that postdates that synthesis (`synthesis-impact.py scan`), it records a candidate here; `knowledge-refresh` push-mode §1 merges these into its topic menu (labelled "newer evidence" vs the time-based "stale"), and `knowledge-finalize` Step 9 clears the entry once a refreshed synthesis lands. Two writers on `knowledge-binding.py`: `add-refresh-candidates` (union/dedup by `synthesis_slug`) and `resolve-refresh-candidate` (remove-on-resolve, keeping the manifest small — clears an entry by `synthesis_slug` **or** by `via_pages[]` citation overlap when `knowledge-finalize` passes `--cites`, so a refresh landing under a divergent slug still resolves it). Pure observability — it never auto-rewrites a synthesis; it only flags and suggests. Pre-0.1.5 bindings have no key, so every reader uses `.get("refresh_candidates", [])`.

`curator_defaults` (added at schema 0.0.3) configures the inverted pipeline's `knowledge-curate` and `knowledge-fetch` phases — see `references/inverted-pipeline.md` and `references/fetch-cache-design.md`. The fetch-cache itself lives at `<knowledge-root>/.cogni-knowledge/fetch-cache/` by convention; the path is derivable from the knowledge root and is therefore not echoed into the binding.

`research_defaults` records the knowledge base's default output config, captured at `knowledge-setup` (Step 2.5) and inherited by every `knowledge-plan` run. Schema 0.1.1 (#309 P1.2-rest) added `market` + `output_language`; schema 0.1.2 (#309 P2) widens it with the four writer-quality knobs `prose_density` / `tone` / `citation_format` / `target_words`. Resolution precedence per key: explicit flag > this block > (the market's registry `default_output_language` for language only) > the hard default (`dach`/`en`/`standard`/`objective`/`ieee`/`4000`). It is an output concern, not a curator one, so it sits as a sibling block — not inside `curator_defaults`. Consumers MUST read it with `.get("research_defaults", DEFAULT_RESEARCH_DEFAULTS)` and each key with a per-key `.get(key, DEFAULT)` so a pre-0.1.2 binding (or a pre-0.1.1 binding with no block at all) falls straight through. Only `knowledge-plan` reads it (as a fallback); `knowledge-curate` continues to read `market` from `plan.json`. The four writer-quality knobs are **flag-or-default** at setup (not prompted — only market + language are prompted, since they lack a safe silent default), and overridable per run via `knowledge-plan --prose-density|--tone|--citation-format|--target-words`.

Schema bumps mirror the data shape, not the plugin tag. The schema goes `0.0.1 → 0.0.2 → 0.0.3` for additive field adds; the jump to `0.1.0` aligned with the `plugin.json` v0.1.0 maturity-flip at M12; `0.1.1` added `research_defaults` (#309 P1.2-rest, v0.1.35); `0.1.2` widened it with the four writer-quality knobs (#309 P2, v0.1.36); `0.1.3` defined the `topic_lineage.covered_themes[]` entry shape for question-node cross-run accumulation (#409, v0.1.52); `0.1.4` added the `charter` block (base-level steering captured at `knowledge-setup` Step 2.5) + seeded `topic_lineage.open_themes[]` with the theme backlog; `0.1.5` added the top-level `refresh_candidates[]` block (evidence-aware refresh signals flagged at `knowledge-ingest-source` time). Consumers that need newer fields on an older binding MUST `.get(..., DEFAULT)` — `cmd_read` returns the binding as-is and does not auto-migrate.

The file is small (< 4 KiB even at 20+ deposited projects). It is *not* a database — it is a narrative manifest that records what the user has chosen to bind together. Search across the wiki itself for content; consult the binding only for "what projects fed this base".

## Delegation contract

The hard rule: **no logic that already exists upstream**. If you find yourself writing code that duplicates a `cogni-wiki` script or a `cogni-research` agent, stop and re-delegate. The full mapping lives in `references/delegation-contract.md`.

A few specifics:

- `knowledge-setup` does NOT re-implement `wiki-setup`. It only handles the `binding.json` half.
- `knowledge-ingest` does NOT dispatch `cogni-wiki:wiki-ingest` — it calls the helper scripts (`backlink_audit.py`, `wiki_index_update.py`) directly at script level (M6 contract).
- `knowledge-distill` (Phase 4.5, #336) makes cogni-knowledge a **locked writer into the wiki tree**: `concept-store.py` performs the concept/entity page read-modify-write under cogni-wiki's canonical `_wiki_lock` (`<wiki-root>/.cogni-wiki/.lock`), which it **imports** from `_wikilib` (via the orchestrator-resolved `--wiki-scripts-dir`) rather than re-inlining — honouring cogni-wiki's "new shared-state writers MUST import `_wiki_lock`" contract. Source pages (Phase 4) are unique-by-construction and need no lock; concept/entity pages are shared-state (a prior run created them) and do.
- `knowledge-resume` computes wiki health **natively** on the vendored `health.py` (resolved vendored-first via `resolve_wiki_scripts()`, FMO Phase 8 d2 re-home, v0.1.87) — it no longer dispatches `cogni-wiki:wiki-resume`, so a standalone Karpathy base shows resume status without cogni-wiki installed (analog to `knowledge-query` / `knowledge-dashboard`).

## Lineage stamping

`derived_from_research: <slug>` in a wiki page's YAML frontmatter is what cycle-guard reads to detect circular evidence without a filesystem walk. In the v0.1.0 inverted pipeline, `knowledge-finalize` sets this field inline when it atomically writes the synthesis page — no separate stamping pass runs. The legacy `lineage-stamp.py` helper (which globbed `wiki/**/*.md` and stamped Mode-A deposits derived from `raw/research-<slug>/`) is archived under `_archive/scripts/` alongside the chain that used it.

## Conventions

- Skill names: `knowledge-*` (generic skill names like `setup`, `research`, `resume` MUST be prefixed per `../CLAUDE.md` §"Contributing").
- Skill frontmatter: `name`, `description`, `allowed-tools` — same shape as cogni-wiki/cogni-research skills.
- Script CLI: `python3 scripts/<name>.py --action ...`, JSON envelope output.
- Path conventions: knowledge bases default to `cogni-knowledge/<knowledge-slug>/` (standard cogni-plugin convention, matching `cogni-wiki/{slug}/`).
- Slug derivation: `_knowledge_lib.slugify` is the single source of truth (source slugs in `knowledge-ingest`, synthesis slug in `knowledge-finalize`). It **transliterates** non-ASCII before stripping (German `ä→ae`/`ö→oe`/`ü→ue`/`ß→ss`, then NFKD de-accent) so `für`→`fuer`, not `f-r` (#303, v0.1.4); empty/non-alnum input → `""` (callers apply a hash fallback). Localized reference headings come from the same module's `ref_heading(lang)` map.
- Index one-liner hygiene (#387, v0.1.39): `_knowledge_lib.sanitize_summary` normalizes stray typographic substitutes an LLM may emit where a space belongs — U+2020/U+2021 daggers (`§†30`, `Dezember†2025`) and the exotic spaces U+00A0/U+202F/U+2009 — back to a regular space before the summary reaches the reader-facing `wiki/index.md` one-liner. Applied at all three `wiki_index_update.py --summary` call sites — `knowledge-ingest` Step 4.2, `knowledge-distill` Step 6.2, `knowledge-finalize` Step 7 — via the env-var `python3 -c` pattern (never interpolate untrusted text into a literal). Unlike `slugify` it is a **display** sanitizer: accents and non-ASCII letters are preserved verbatim, no transliteration. Orthogonal to the #324 `--max-summary 240` mid-word clamp (both stay).
- Versioning: bump patch on any skill/script change; mirror in `marketplace.json` (`../CLAUDE.md` §"Version Management").
- Citation count (F24 pin): the one authoritative count is `len(.metadata/citation-manifest.json::citations)` for the latest draft version — surfaced by `pipeline-summary.py project`. The composer's return `citations` field and the `knowledge-compose`/`knowledge-verify` summary lines all source this same length (never an LLM estimate). A `verify-vN.json` `counts.total` is verdicts scored for draft-vN at that round's start (before the §3.2 `sentence_not_in_draft` prune), NOT the citation count.
