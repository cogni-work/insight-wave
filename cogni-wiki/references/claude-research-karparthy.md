# Karpathy's LLM Wiki: the case for compiled knowledge over retrieval

Andrej Karpathy's "LLM Wiki" is a pattern for building persistent, LLM-maintained knowledge bases from plain markdown files — an explicit alternative to Retrieval-Augmented Generation (RAG) for personal and small-team knowledge work. Published as a GitHub Gist on April 4, 2026 (one day after a viral tweet that reached **17 million views**), the concept argues that RAG's fundamental flaw is amnesia: every query forces the LLM to rediscover knowledge from scratch, with no accumulation across sessions. The wiki pattern flips this by having the LLM **compile** raw sources into structured, interlinked articles at ingestion time rather than retrieving chunks at query time. Karpathy's own research wiki grew to roughly **100 articles and 400,000 words**, navigated via a simple index file with zero vector database infrastructure. The concept ignited one of 2026's most contested AI architecture debates, with the gist collecting 5,000+ stars and spawning 15+ community implementations within a week.

## The architecture: three layers and a compiler analogy

The LLM Wiki separates knowledge into three explicit layers. **Layer 1 (raw/)** holds immutable source documents — PDFs, articles, web clips, transcripts. The LLM reads these but never modifies them; they remain the ground truth. **Layer 2 (wiki/)** is a directory of LLM-generated markdown files: entity pages, concept summaries, comparisons, and syntheses. The LLM owns this layer entirely, creating and updating pages as new sources arrive. **Layer 3 (the schema)** is a configuration document (CLAUDE.md for Claude Code, AGENTS.md for Codex) that defines conventions, page structures, and workflows — turning the LLM into "a disciplined wiki maintainer rather than a generic chatbot."

Karpathy's metaphor is deliberate: **"Obsidian is the IDE, the LLM is the programmer, and the wiki is the codebase."** Raw sources are source code; the LLM compiles them into a structured artifact; lint operations serve as tests; queries are runtime. This framing reveals the core innovation — heavy processing happens once at ingestion, not repeatedly at query time.

Four operations define the workflow. During **ingest**, the LLM reads a new source, discusses takeaways with the user, writes a summary page, and cascades updates across **10–15 existing wiki pages**, updating cross-references and flagging contradictions. During **query**, the LLM reads the `index.md` first (a lightweight catalog of all pages with one-line summaries), identifies relevant articles, then reads only those — a process Karpathy notes works "surprisingly well" at moderate scale without embedding infrastructure. The **lint** operation is a periodic health check: scanning for contradictions, orphan pages, stale claims, and concepts referenced but lacking their own page. Finally, **file-back** closes the loop: valuable query outputs (comparisons, analyses, connections) get saved as new wiki pages, so explorations compound alongside ingested sources.

The tooling stack is intentionally simple. Obsidian provides the viewing interface (graph view, web clipper, local-first storage). Claude Code or similar agents do the maintenance. Git provides version control for free. For larger wikis, Karpathy recommends **qmd** — a local hybrid BM25/vector search engine for markdown files built by Shopify CEO Tobi Lutke — but emphasizes that most wikis won't need it.

## How it diverges from RAG at the architectural level

RAG and LLM Wiki solve the same problem — giving LLMs access to private knowledge — but through fundamentally different mechanisms. RAG follows a six-step pipeline: chunk documents into **256–1,024 token fragments**, embed them as vectors, store them in a vector database, embed the user's query at runtime, retrieve the top-K most similar chunks, and inject them into the context window. The LLM then synthesizes an answer from these fragments. Every query starts fresh. Nothing compounds.

The LLM Wiki eliminates the entire embedding-retrieval pipeline. Knowledge is pre-synthesized into human-readable markdown. Cross-references exist before anyone asks a question. Contradictions are flagged during ingestion, not discovered (or missed) during retrieval. The result is a **stateful system** where each new source makes the entire knowledge base richer, versus RAG's **stateless** query-time assembly.

Three technical differences matter most. First, **retrieval reliability**: RAG depends on embedding similarity, which can silently miss semantically relevant content ("cancellation policy" may not match "ending your subscription"). The wiki approach loads pre-synthesized articles directly — if the content exists, the LLM sees it. Second, **synthesis quality**: RAG forces the model to reconcile disparate fragments on every query, paying what one analyst called a "synthesis tax" repeatedly. The wiki pays that tax once at ingestion. Third, **debuggability**: when a RAG system gives a wrong answer, tracing which chunks were retrieved and why requires reverse-engineering opaque vector math. With the wiki, every claim traces to a specific markdown file that a human can read.

The token efficiency difference is real but nuanced. Against **naive full-document loading**, a curated wiki can reduce token usage by up to **95%** — a 20,000-token policy document becomes a 2,000-token summary. Against **optimized RAG** (which typically injects 2,000–5,000 tokens of context per query), the per-query difference narrows significantly. The wiki's advantage is structural: the same synthesized knowledge serves every future query without re-retrieval costs.

## Where LLM Wiki wins — and where RAG remains essential

The LLM Wiki excels in a specific zone: **bounded knowledge bases under roughly 50,000–100,000 tokens of compiled content**, where knowledge accumulates over time and synthesis across sources matters more than raw retrieval speed. Karpathy lists five use cases explicitly: personal self-improvement tracking, deep multi-month research projects, reading companions for books, team knowledge bases fed by Slack and meeting transcripts, and general knowledge accumulation (competitive analysis, course notes, hobby research).

A community benchmark tested both approaches head-to-head on a **production codebase (~50K lines of code)** across 130 questions in 7 tournaments. The wiki **significantly outperformed RAG on questions about deleted or archived logic** — it maintained institutional memory that had physically disappeared from the repository. The combined approach (wiki for context, RAG for verification) **never lost a single round**, even on tasks designed to favor RAG.

RAG remains the correct architecture in four scenarios. **Scale**: when the knowledge base exceeds hundreds of thousands of documents, the index file itself overflows the context window, and the ingestion cost becomes prohibitive. **Dynamic content**: for data that changes daily (stock prices, inventory, news feeds), maintaining a curated wiki is impractical. **Source attribution**: RAG natively carries chunk-level provenance (document, page, URL), which is critical for legal, academic, and compliance applications. **Multi-domain enterprise**: when a single assistant spans HR, finance, IT, and legal with role-based access controls, RAG's infrastructure handles the complexity that markdown folders cannot.

The most practical insight from the analysis is that **the hybrid approach dominates both standalone methods**. Stable, foundational knowledge lives in a curated wiki loaded as system context. Dynamic, large-scale, or user-specific content gets retrieved via RAG. The wiki can even serve as a routing layer — its index tells the agent which RAG queries to run. Several enterprise analysts noted that this hybrid mirrors what well-governed data catalogs already provide: curated metadata summaries (the "wiki") sitting atop raw data stores (the "RAG corpus").

## The real risks: persistent errors and closed epistemic loops

The most incisive criticism came not from those who dismissed the concept but from those who engaged deeply with its implications. A commenter on Karpathy's gist framed the core tradeoff precisely: **"We're trading ephemeral hallucinations for persistent errors."** In RAG, a hallucination affects one query and then vanishes. In the wiki, a misinterpretation during ingestion gets baked into a page, which informs future queries, which may generate new pages building on the error. Knowledge compounds — but so do mistakes.

This connects to the **closed epistemic loop** problem, identified by aiHola's Oliver Senti as the most under-discussed risk. When the same LLM compiles the wiki, maintains the wiki, and queries the wiki, there is no external verification layer. The lint operation catches structural issues (broken links, orphan pages) but not factual errors. A production user running the pattern for 12+ months confirmed: "A bad wiki article becomes a prior that poisons future generations. You need a consolidation engine that scores, merges, and prunes — not just appends."

Other practical limitations compound at scale. **Lossy compression** is inherent: the wiki is a summary, and summaries discard edge cases, exact wording, and subtle distinctions. **Cascading updates** create fragility — one correction can ripple across dozens of interconnected pages, and expecting the LLM to maintain consistency across hundreds of articles is "risky," as critic Mehul Gupta argued. **Token costs** for ingestion are unpublished; each new document touches 10–15 pages with full read-and-rewrite operations, and nobody running the pattern in production has shared real cost figures. **Concurrency** breaks without transactional database support — multiple agents writing simultaneously create race conditions that plain markdown folders cannot handle. And the **ingestion gap** is binary: community testing found that a "mostly finished" wiki performed **17% worse** than a fully compiled one, suggesting the system is fragile during transitions.

The enterprise case is particularly weak. Epsilla's analysis called the approach "a brilliant prototype for a single user" but noted that a mid-sized enterprise deals with millions of documents, requires role-based access control and audit trails, and cannot accept a knowledge base that exists as "a set of easily copied text files." The security model is nonexistent. The scalability ceiling is real. The governance layer is absent.

## A concept that named a real frustration

The community reaction reveals why the idea resonated beyond its technical merits. As aiHola's analysis noted, "The reason Karpathy's post went viral is that he named a frustration most power users share: the feeling that every conversation with an LLM starts from zero." The concept struck a nerve because it articulated something millions of users felt but hadn't formalized — that chat-based AI interaction is fundamentally wasteful, with good answers disappearing into scroll-back history.

The "idea file" distribution model may prove more significant than the wiki architecture itself. Karpathy explicitly chose not to share code: "In this era of LLM agents, there is less of a point/need of sharing the specific code/app, you just share the idea, then the other person's agent customizes & builds it." This represents a shift from open source to **open ideas** — distributing intent rather than implementation, and letting each user's agent materialize the specifics. Community members extended this further, using gists as agent-to-agent communication channels, pushing diagrams and context between different AI frontends.

The philosophical critique also deserves attention. HN user *qaadika* argued that "the bookkeeping Karpathy outsources — filing, cross-referencing, summarizing — is where genuine understanding forms." There is a meaningful difference between an LLM-authored research index and a genuine second brain. The wiki pattern automates maintenance brilliantly, but whether outsourcing the synthesis step to an LLM builds understanding or merely creates the illusion of it remains an open question. Karpathy's own workflow — staying involved during ingestion, reading summaries, guiding emphasis — suggests he recognizes this tension. The highest-fidelity version of the pattern keeps humans firmly in the curation loop.

## Conclusion

Karpathy's LLM Wiki is neither the death of RAG nor merely "RAG with extra steps." It is a **compile-time knowledge architecture** that trades infrastructure complexity for curation effort, and query-time processing for ingestion-time synthesis. Its sweet spot is clear: individual researchers and small teams with bounded, accumulating knowledge bases where cross-document synthesis matters more than raw scale. The strongest evidence comes from the head-to-head benchmark showing that the combined wiki-plus-RAG approach outperformed either method alone — suggesting the real lesson is not replacement but **layering**. The persistent-error risk is genuine and unsolved, the enterprise case is premature, and the scalability ceiling is real. But the core insight — that knowledge should compound rather than be rediscovered on every query — represents a legitimate architectural principle that will likely influence how LLM systems handle memory for years to come. The Memex that Vannevar Bush imagined in 1945 finally has its librarian. Whether that librarian can be trusted to work unsupervised is the next question to answer.
