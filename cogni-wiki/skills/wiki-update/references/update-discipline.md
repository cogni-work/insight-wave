# Update Discipline — Worked Examples

Two realistic scenarios: a contradiction fix and a new-source refinement. Both walk through the diff-before-write discipline end-to-end.

## Example 1: Contradiction fix

### Setup

`wiki-lint` reported:

> 🟡 `[[rlhf-overview]]` claims "RLHF uses human preference labels." `[[constitutional-ai]]` claims "RLHF labels can be generated by a separate critic model." These contradict each other on what RLHF *is*.

The user asks: "Reconcile these two pages."

### Step 1: Read both pages

`[[rlhf-overview]]` frontmatter: `type: concept`, `sources: [../raw/ouyang-et-al-2022-instructgpt.pdf]`. Body defines RLHF as "Reinforcement Learning from Human Feedback — using human-generated preference labels to train a reward model."

`[[constitutional-ai]]` frontmatter: `type: concept`, `sources: [../raw/bai-et-al-2022-constitutional-ai.pdf]`. Body says "Constitutional AI replaces RLHF's human labels with AI-generated critiques, making the 'feedback' in RLHF come from a separate critic model rather than humans."

### Step 2: Diagnose the contradiction

The two pages are not actually contradicting each other — they are using "RLHF" at different levels of abstraction. `[[rlhf-overview]]` uses the original Ouyang et al. definition where human labels are load-bearing. `[[constitutional-ai]]` uses a generalized sense where any preference signal (human or AI) counts as "feedback."

The fix is a clarification, not a reversal. The right resolution:

- Tighten `[[rlhf-overview]]` to say "RLHF as introduced by Ouyang et al. 2022 uses *human-generated* preference labels"
- Tighten `[[constitutional-ai]]` to say "Constitutional AI generalizes the RLHF pipeline by replacing the human labeler with an AI critic model — the same reward-model training loop, a different source of preferences"

Both changes are additive (scoping the claims more tightly) and both cite their respective sources.

### Step 3: Diff before write — `[[rlhf-overview]]`

State to the user:

> **Current** (line 12):
>   "RLHF — Reinforcement Learning from Human Feedback — uses human-generated preference labels to train a reward model."
>
> **Proposed**:
>   "RLHF as introduced in Ouyang et al. 2022 [InstructGPT] uses *human-generated* preference labels to train a reward model. Later work (e.g. [[constitutional-ai]]) generalizes this pipeline to non-human preference sources."
>
> **Why**: scopes the original definition to its source, then links to the generalization. Preserves the original meaning, adds precision, resolves the reported contradiction without weakening either page.

### Step 4: Apply via Edit

```
old_string: "RLHF — Reinforcement Learning from Human Feedback — uses human-generated preference labels to train a reward model."
new_string: "RLHF as introduced in Ouyang et al. 2022 [InstructGPT] uses *human-generated* preference labels to train a reward model. Later work (e.g. [[constitutional-ai]]) generalizes this pipeline to non-human preference sources."
```

Update `updated:` to today.

### Step 5: Diff before write — `[[constitutional-ai]]`

Mirror process on the other page. Tighten its RLHF reference to match the scoped definition.

### Step 6: Related sweep

`grep -rl "RLHF" wiki/` returns 8 pages. For each, read the paragraph containing "RLHF" and check whether the narrower definition breaks anything. Usually it doesn't — most mentions are informal. Note any pages that now have mild ambiguity by adding `## Stale (2026-04-12)` at the top with a one-liner.

### Step 7: Log

```
## [2026-04-12] update | rlhf-overview — contradiction
## [2026-04-12] update | constitutional-ai — contradiction
```

### Step 8: Report

> Reconciled the RLHF definition across `[[rlhf-overview]]` and `[[constitutional-ai]]` by scoping the original definition to Ouyang 2022 and linking forward to the CAI generalization. Swept 8 related pages — no follow-up edits needed. Run `wiki-lint` next to confirm the contradiction warning is cleared.

## Example 2: New-source refinement

### Setup

`wiki-ingest` was invoked on `raw/chen-et-al-2024-rlhf-scaling.pdf`, which significantly refines what is already on `[[rlhf-overview]]`. `wiki-ingest` detected the collision and passed control here.

### Step 1: Read page and source

Current `[[rlhf-overview]]` claims "Reward-model quality saturates around 1B parameters." The new source shows this was measurement noise — the saturation curve extends to at least 70B with diminishing but non-zero returns.

### Step 2: Diff before write

> **Current** (line 34):
>   "Reward-model quality saturates around 1B parameters, beyond which additional scale yields no measurable gains."
>
> **Proposed**:
>   "Earlier work suggested reward-model quality saturates around 1B parameters, but Chen et al. 2024 [../raw/chen-et-al-2024-rlhf-scaling.pdf] show the saturation claim was measurement noise — quality continues to improve (with diminishing returns) to at least 70B."
>
> **Sources to add**: `../raw/chen-et-al-2024-rlhf-scaling.pdf`
>
> **Why**: preserves the historical claim (with attribution to the earlier era), adds the refinement with citation. The original source remains cited so readers can trace both positions.

### Step 3: Apply via Edit

Edit the body paragraph. Append the new source to the `sources:` frontmatter list. Update `updated:`.

### Step 4: Related sweep

Any page that cited `[[rlhf-overview]]`'s saturation claim is now indirectly wrong. Sweep them and either update them inline or mark them stale.

### Step 5: Log and report

```
## [2026-04-12] update | rlhf-overview — new-source
```

> Updated `[[rlhf-overview]]` with Chen et al. 2024 — reward-model scaling no longer saturates at 1B. Swept 3 dependent pages; 2 needed follow-up edits, 1 is now marked `## Stale`. Consider running `wiki-lint` to confirm no broken claims remain.

## Anti-patterns

- **"Update" that rewrites the whole page from scratch.** Use Edit, not Write. Preserve unchanged bytes.
- **"Update" that adds a claim without a source.** Stop and ask for the source.
- **"Update" that deletes the old claim without explanation.** Mark superseded content explicitly — "Earlier work suggested…" — rather than erasing it.
- **"Update" that fixes one page and leaves contradictions on related pages.** Always sweep.
- **"Update" that changes `id:`.** Never. The `id` is immutable.
