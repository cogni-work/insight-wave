# Theme Manifest (v1.0)

`themes/<slug>/manifest.json` describes which tiers a theme exposes вЂ” tokens,
assets, components, templates вЂ” so consumer skills (e.g. `pick-theme`,
`render-html-slides`) can resolve the right files without grepping a theme
directory at every call site.

The manifest is **optional**. Themes that ship only `theme.md` (the v1
format) are valid and treated as **tier-0**. The validator passes them as a
no-op.

The schema is `references/theme-manifest.schema.json` (JSON Schema draft-07).
The validator is `scripts/validate-theme-manifest.py` (Python stdlib only).

For the end-to-end tier-0 в†’ tiered walkthrough (when to migrate, tier-by-tier
authoring, validation, rollback, common pitfalls), see
[`docs/theme-system-v2-migration.md`](../docs/theme-system-v2-migration.md).

## Locked design decisions (epic #132 resolutions)

| Question | Decision |
|----------|----------|
| Token format | JSON canonical (`tokens/colors.json`, `typography.json`, `spacing.json`, `radii.json`, `shadows.json`, `motion.json`); `tokens/tokens.css` generated by `scripts/generate-tokens-css.py` |
| Token parity | If both JSON and CSS exist, the CSS must match the regenerated output. Drift is a hard validation failure. |
| Component versioning | `live` and `live_within_session` are reserved keys вЂ” schema rejects them. Phase 3 will define semantics. |
| Component format | HTML primary (`.html`); JSX permitted alongside (`.jsx`); other extensions rejected in v1.0. |
| Voice / copy | No `copy` tier in v1.0 вЂ” voice rules stay in `theme.md`. The `copy` key is reserved at any depth. |

The validator enforces the reserved-key reservations recursively at any
nesting depth, not just at the manifest root.

## Required fields

| Field | Type | Notes |
|-------|------|-------|
| `schema_version` | string, const `"1.0"` | Locked; bump to `"1.1"` only when the schema changes. |
| `name` | string, в‰Ґ 1 char | Human-readable name. |
| `slug` | string, kebab-case | Must match the theme directory name. |

## Optional fields

| Field | Type | Notes |
|-------|------|-------|
| `tiers.tokens` | string | Path to the tokens directory (typically `"tokens/"`). |
| `tiers.assets` | string | Path to the assets directory. |
| `tiers.components.{web,deck,dashboard}` | string | Path to a surface-specific component directory. |
| `tiers.templates.{deck,report,landing,dashboard}` | string | Path to a template file. |
| `showcase` | string | Path to a preview artifact (JSX, HTML, PPTX). |
| `voice_ref` | string | Anchor into `theme.md`, e.g. `"#voice--copy-guidelines"`. Not a filesystem path. |

A theme with an empty `tiers: {}` is structurally valid; the validator only
fails when a declared tier path does not exist on disk.

  "slug": "sample-tokens",

The validator checks that `tokens/` exists, then regenerates `tokens.css`

## Reserved keys

The keys `live`, `live_within_session`, and `copy` MUST NOT appear anywhere
in the manifest in v1.0 вЂ” not at the root, not inside `tiers`, not inside
`components`. The validator scans recursively and fails on the first
occurrence with the path to the offending key.

If a future phase needs these keys, bump `schema_version` and update the
schema; the recursive scan is keyed on the v1.0 set only.

## Running the validator

```bash
# Tier-0 themes (no manifest.json) вЂ” should always pass:
python3 cogni-workspace/scripts/validate-theme-manifest.py cogni-workspace/themes/cogni-work
python3 cogni-workspace/scripts/validate-theme-manifest.py cogni-workspace/themes/_template

# Tiered theme:
python3 cogni-workspace/scripts/validate-theme-manifest.py cogni-workspace/themes/<slug>
```

Output is the standard cogni-workspace JSON envelope:

```json
{"success": true, "data": {"tier": "tiered", "name": "...", "slug": "...", "tiers_present": [...]}, "error": ""}
```

On failure, `success` is `false` and `error` names the first failure with a
JSON-Pointer-style path.

## Out of scope for v1.0

- **Live-resolution semantics** for `live` / `live_within_session` (Phase 3).
- **`copy` tier** for voice/copy templates (Phase 3).
- **CI integration** of the validator вЂ” manual invocation as a pre-PR check
  is sufficient for v1.0.
- **Strict HTML5 conformance** for component primitives вЂ” `html.parser`
  parseability is enforced at the manage-themes authoring level (#128); the
  schema only enforces the directory-shape contract.
